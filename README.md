# CS250
Throughout this course, I learned that interpreting user needs starts with listening carefully and breaking large ideas into smaller, manageable pieces. Creating user stories helped me translate general requests into clear, actionable tasks. Instead of thinking about a feature in technical terms right away, I learned to frame it from the user’s perspective first. That made it easier to understand the purpose behind the work and implement it more effectively. User stories also helped connect requirements, testing, and development so that everyone stayed aligned on what “done” actually meant.

When approaching program development, I now think more in terms of iteration and communication rather than trying to build everything perfectly on the first attempt. The Agile process reinforced the value of feedback, incremental improvement, and adapting when requirements change. Going forward, I want to continue using practices like writing clear user stories, breaking work into smaller tasks, and regularly reviewing progress. Being a good team member in software development means communicating openly, asking questions when something is unclear, and being willing to adjust. This project showed me that collaboration and clarity are just as important as technical skill in producing strong results.
